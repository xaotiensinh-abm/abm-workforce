# Goal

> **Use this skill when:** implementing features using TDD (test-first approach)

Áp dụng kỷ luật Test-Driven Development (TDD) khắt khe cho mọi đối tượng tính năng mới, bugfix và behavior change. Code production phải bảo đảm được tạo nên sau khi test đã Failing (Đỏ).

# Instructions

1. **RED - Viết Failing Test:**
   - Viết MỘT test case cho MỘT hành vi mong muốn trong thực thi. Dùng code thực (hạn chế mock unless bất khả kháng).
   - TIẾN HÀNH CHẠY TEST ĐỂ THẤY NÓ FAIL. Chắc chắn nó fail đúng lý do tính năng thiếu (không phải typo hay lỗi setup). Bước này KHÔNG ĐƯỢC BỎ QUA.
2. **GREEN - Viết Minimal Code:**
   - Viết đủ lượng code tóm gọn nhất để pass test. KHÔNG viết dư tính năng, KHÔNG over-engineer tiên đoán. (YAGNI).
   - TIẾN HÀNH CHẠY LẠI TEST, xem nó Pass (Green) rực rỡ và các test bên cạnh không bị vỡ.
3. **REFACTOR - Dọn dẹp code:**
   - Vẫn duy trì green test, clean code, loại lặp và tu sửa tên các biến/hàm.
4. Lặp lại cho chu kỳ Requirement khác.
5. Xem file `testing-anti-patterns.md` để phòng tránh các lỗi test phổ biến.

# Examples

- **RED:** `test('rejects empty email', async () => { ... expect(result.error).toBe('Email') })`. Run -> FAIL.
- **GREEN:** Thêm cấu trúc block `if (!email) return { error: 'Email' }`. Run -> PASS.
- **REFACTOR:** Trích xuất Error thành constant Enum. Run -> PASS.

# Constraints

- 🚫 KHÔNG VIẾT CODE PRODUCTION nếu chưa có test báo FAIL ở cùng terminal trước đó. Sunk-cost hay reference cũng cấm tận dụng (delete nó và làm từ Test qua).
- 🚫 Không sửa code để ôm đồm "hi vọng nó pass luôn cả cục".
- Không nản lòng, TDD loại bỏ bug khó, gỡ lỗi nhanh mạnh hơn debug sau build.
