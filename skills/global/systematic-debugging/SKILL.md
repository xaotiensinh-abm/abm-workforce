# Goal

> **Use this skill when:** debugging any bug, test failure, or unexpected behavior

Tiếp cận và sửa lỗi một cách có hệ thống, không sửa rải rác hay dự đoán vô căn cứ. Luôn tìm ra nguyên nhân gốc rễ (root cause) trước khi viết code sửa lỗi. Fix triệu chứng (symptoms) là biểu hiện của thất bại.

# Instructions

1. **Giai đoạn 1: Khám phá nguyên nhân gốc rễ (Root Cause Investigation)**
   - Đọc kỹ Error Messages. Tái tạo lỗi một cách nhất quán.
   - Kiểm tra các thay đổi gần đây (git diff).
   - Thêm log/diagnostic layer nếu hệ thống có nhiều component.
   - Trace dữ liệu ngược lên đầu nguồn. KHÔNG ĐỀ XUẤT FIX TẠI BƯỚC NÀY.
2. **Giai đoạn 2: Phân tích mẫu (Pattern Analysis)**
   - Tìm code đang hoạt động tốt hoặc implementations tham chiếu, để so sánh sự khác biệt chi tiết.
3. **Giai đoạn 3: Đặt giả thuyết và Kiểm chứng (Hypothesis & Testing)**
   - Viết ra giả thuyết rõ ràng ("Lỗi do X vì Y").
   - Đưa ra kiểm thử xác minh giả định với sự thay đổi nhỏ nhất.
4. **Giai đoạn 4: Thực thi (Implementation)**
   - Viết test case tự động sẽ FAIL nếu chưa fix (TDD style).
   - Triển khai MỘT sửa đổi duy nhất. Không đan xen refactor.
   - Xác minh lại sự cố thay đổi qua Test.
   - Nếu thử 3 lần fix mà vẫn fail -> DỪNG LẠI. Phải rà soát lại kiến trúc hệ thống, trao đổi với user. Lỗi đã trầm trọng.

# Examples

- **Khi gặp lỗi `undefined` ở hàm `processData`:**
  - KHÔNG ghi ngay `if (!data) return;` (hack fix symptoms).
  - Trace ngược xem ai gọi hàm đó và truyền `undefined`. Sửa ở layer gọi dữ liệu sai.
- Nếu config không chạy, hãy test env variable có thật sự đi vào build chưa.

# Constraints

- 🚫 KHÔNG dự theo linh cảm mò mẫm, KHÔNG "thử fix nhanh xem sao".
- 🚫 KHÔNG thêm nhiều thay đổi cùng lúc rồi chạy test hên xui phán đoán.
- Phải dùng kỹ thuật truy xuất dữ liệu ngược (root-cause-tracing).
