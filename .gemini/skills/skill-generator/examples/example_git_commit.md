# Goal

Sinh commit message chuẩn Conventional Commits từ staged changes,
đảm bảo nhất quán và dễ trace trong lịch sử Git.

# Instructions

1. Chạy `git diff --cached --stat` → xem danh sách files staged.
2. Chạy `git diff --cached` → đọc nội dung thay đổi chi tiết.
3. Phân tích thay đổi → xác định type phù hợp nhất:
   - **feat** = tính năng mới
   - **fix** = sửa lỗi
   - **docs** = chỉ tài liệu
   - **style** = format, không thay đổi logic
   - **refactor** = tái cấu trúc, không thêm feature hay fix bug
   - **test** = thêm/sửa test
   - **chore** = bảo trì (deps, config, CI)
4. Xác định scope — module/component chính bị ảnh hưởng.
   - Nếu thay đổi ≤ 3 files cùng module → scope = tên module
   - Nếu thay đổi > 3 files hoặc cross-module → bỏ scope
5. Viết description ≤ 72 ký tự, bắt đầu bằng chữ thường.
6. Nếu breaking change → thêm `!` sau type: `feat!: description`
7. Đề xuất: `git commit -m "<type>(<scope>): <description>"`

# Examples

## Ví dụ 1: Thêm tính năng đăng nhập
**Input:** Staged: `src/auth/login.ts`, `src/auth/login.test.ts`
**Output:** `git commit -m "feat(auth): add login functionality with unit tests"`

## Ví dụ 2: Sửa lỗi giao diện
**Input:** Staged: `src/components/Header.tsx` (sửa 3 dòng CSS)
**Output:** `git commit -m "fix(ui): resolve header alignment issue on mobile"`

## Ví dụ 3: Cập nhật tài liệu
**Input:** Staged: `README.md`, `docs/setup.md`
**Output:** `git commit -m "docs: update setup instructions and readme"`

# Constraints

- Dòng commit đầu tiên ≤ 72 ký tự — Git và GitHub truncate nếu dài hơn
- Bắt đầu description bằng chữ thường — convention chuẩn, không capitalize
- Không kết thúc bằng dấu chấm — tiết kiệm ký tự, convention chuẩn
- Dùng tiếng Anh cho description — trừ khi user yêu cầu tiếng Việt
- Chọn type chính xác nhất — nếu vừa feat vừa fix, tách thành 2 commits
